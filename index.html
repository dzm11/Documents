A design system your agents can actually read.

A concise field manual, synthesised from seven essays by practitioners at zeroheight and Southleft, for teams who want their design system to be legible to coding agents — not just to humans.

Tokens Contracts MCPs & Skills Schemas Diagnostic layer Declared docs
Why this exists Most design systems were written for humans who can ask a colleague when something is unclear. Coding agents cannot. They take whatever you give them and run with it. This manual collects what to fix, in the order it matters, so that an agent can compose your product without guessing.

Design systems were always a means to an end. We didn't build token architectures for the joy of naming colours; we built them so that products could be shipped faster, more consistently, and at greater scale.01 The end has just changed shape. The next products your system needs to support are assembled by agents, in the moment, from your component inventory — and the margin for error in that output is functionally zero.01

That is why every "we have a design system, but…" story now matters more than it used to.04 A messy system used to be a productivity problem. Today it is a strategic liability: a human developer can compensate for inconsistent tokens or stale docs; an agent cannot.04

The good news, repeated across all seven essays, is that the fix is unglamorous and well-known. It is the boring stuff: tokens, contracts, schemas, machine-readable documentation, and a diagnostic layer that keeps design and code in sync.01 Do that work in the right order and you do not just survive the shift — you become the team that gets to use it.

The eight steps
  1. 01

    Treat the system as an API, not a gallery.

    An API doesn't hand you a picture and ask you to figure out what it meant. It returns structured data with a contract.06

    The single most important shift is documenting your system as declared data rather than designed artefact. Designed frames — visual layouts, do/don't pairs, callouts pinned around a component — are optimised for human browsing, but force an agent to infer meaning from a node tree.06 Declared documentation lives in description fields, in markdown, in structured data, and an agent can parse it without guessing.

    • Populate component description fields with clear markdown: purpose, usage, constraints, do/don't expressed as text.
    • Couple documentation to components — rename or delete the component and the docs travel with it.06
    • Keep designed frames for human stakeholders, but stop treating them as machine-readable.
    • The readability test: can an agent understand this without visual inference? If not, declare it.
  2. 02

    Fix the foundation before you generate anything.

    For AI to generate a correct, branded, accessible interface, it needs absolute clarity about what your components are, how they behave, and what constraints they must respect.01

    A design system that is "mostly documented" or "good enough for the team that built it" is not enough. Complete is the bar.01 The chain is unforgiving: you cannot generate adaptive UI without a system an agent can reliably consume; you cannot have a consumable system without a diagnostic layer that keeps design and code in sync; you cannot build that layer without first fixing the drift that caused it.01

    • Audit tokens: are they consistent, semantic, and free of one-off hex values in production code?04
    • Write component contracts that specify every prop, state, interaction pattern, and accessibility requirement.01
    • Add metadata that a machine can parse — not just human-readable Figma annotations.
    • Check platform parity: a component locked to one platform is half a contract.01

    Two real failure modes from enterprise teams: designers building careful Figma variants while engineers ship inline styles and hardcoded values; and a 50-engineer team colour-picking from screenshots because the design-to-code path didn't exist.04

    The root cause was the same in both: no structural communication encoded in the system itself.

  3. 03

    Encode the why, not just the how.

    LLMs are trained on generic code and design practices. The choices already represented in your system — and not encoded anywhere — are exactly what they get wrong.02

    Most component docs describe mechanics: what props exist, the difference between a checkbox and a radio. That is the how. What an agent needs in addition is the why: when to use a switch instead of a checkbox, when never to use a semantic-red chip (e.g. to indicate medical risk status), when a "select all / none" pattern is required.02 Those judgments already exist in your team — they just aren't encoded.

    • Treat design guidelines as a new deliverable, shipped alongside components and reference docs.02
    • Write them as agent skills: short, decision-oriented, product-specific.
    • Include negative examples — "do not generate X for Y context" — because the default training data will get this wrong by default.02
  4. 04

    Put MCPs, AGENTS.md, and skills in the codebase.

    When LLMs have access to your entire codebase, they'll use anything they find as a pattern to copy. The models aren't trained on your code; they're trained on public code, which is why Tailwind and ShadCN are so over-represented.02

    Without explicit guardrails, agents will faithfully reproduce whatever's most common in their training set — not what's in your system. Three artefacts close the gap:

    • MCP server — exposes the design system to agents as a structured resource they can query, not scrape.
    • AGENTS.md — a manifest at the root of the repo telling agents which components to use, which patterns are forbidden, where the tokens live.
    • Agent skills — small, focused instruction files that capture decision rules (the why from step three).02
    # AGENTS.md (excerpt)
use_components_from: "@acme/ds"
forbid:
  - "inline tailwind utility classes for colour"
  - "recreating components that already exist in @acme/ds"
tokens: "./tokens/semantic.json"
skills: "./.skills/"
  5. 05

    Ship components as data — not just code.

    What if a Payment element is not a single component, but a set of structured relationships that describe how to use all the individual atomic components?02

    Patterns have always been the design-system bogeyman: built for one product purpose, made of many sub-components, and exploding in variation. Encoding them as rigid code components creates a maintenance burden and pushes teams toward workarounds. 02 Schemas — "components as data" — describe the relationship instead: which components compose into which patterns, what context a component requires, which siblings are valid.

    • For each pattern, ship a schema describing its sub-components, contexts, and allowed siblings — not a monolithic code component.
    • Use semantic naming so an agent can compose meaningfully (a CancelButton in modalContext is far clearer than a generic Button).02
    • This is what enables multi-turn, dynamically recomposing UI — the same primitive that lets products like Expedia render inside ChatGPT.02
  6. 06

    Build the diagnostic layer.

    No feedback loop. No diagnostic layer. Just two teams working in parallel, producing divergent results.04

    Drift is the check engine light of design systems. It rarely fails catastrophically; it just quietly diverges, sprint by sprint, demoralising the people who care most about getting it right.04 The fix is a layer that constantly compares what was designed against what shipped — and surfaces the gap as a first-class signal, not a slack thread.

    • Automated parity checks between Figma source-of-truth and production code (tokens, components, props).
    • Bidirectional sync — when a designer changes a variant, code knows; when an engineer ships a divergence, design sees it.
    • Treat drift as technical debt with a measurable cost, not a vibes problem.
  7. 07

    Put LLMs to work on the unglamorous tasks first.

    Right now, the best way to use LLMs in your design system is to focus on meaningfully filling gaps. Think less about generating new components and more about delivering on the promise of design systems.03

    Don't ask the LLM to invent your input component. Use it for the work that almost never gets prioritised against product roadmaps but quietly erodes trust in the system.03

    • Refactors and migrations — token renames, API changes, deprecations. Glorified find-and-replace at scale.03
    • Interaction prototyping — let designers exercise real flows in the browser via tools like v0, Lovable, or Dessn, surfacing edge cases before engineering starts.03
    • Keeping examples fresh — Storybook stories, doc snippets, sandbox states that go stale the moment a component changes.03

    Don't let an LLM write your design-system documentation. The output reads like a third-rate style guide ("An input is an element users can type into") and lacks the context your team actually needs.03

  8. 08

    Build in the open. Extend, don't restart.

    We need to stop creating new systems for every project. Build within existing ecosystems. Avoid starting fresh unless absolutely necessary.05

    AI thrives in open environments because it has been trained on them. Open code, open documentation, open patterns: the more your system looks like an extension of widely-known primitives, the more predictable agent output becomes.05 The corresponding warning is sameness — when every team uses the same kits and the same prompts, diversity collapses and the design system stops carrying any identity.05

    • Adopt or extend existing systems gradually, component by component.05
    • Make transparency and machine-readability architectural principles from day one.
    • Contribute back when you patch or extend — it keeps the ecosystem (and the models trained on it) honest.
    • Curate against "vibe-coded sameness": where does your product actually need to look unlike the default?
Interlude · 01

The roles move upstream.

If agents can compose interfaces, what happens to the people who used to compose them? The honest answer from across the essays: the work doesn't shrink, it relocates.0105

From

Building individual screens, individual components, individual stories. Picking colours, drawing variants, shipping pixel mocks across to engineering. Performing repetitive system maintenance by hand.05

→
To

Authoring the tokens, contracts, and decision rules that the system uses to assemble itself. Defining the vocabulary, not arranging the sentences. Reviewing, refining, and governing what agents produce.0105

Designers become rule-authors.

Instead of designing individual screens, designers define the vocabulary, the constraints, the rules that govern how screens get generated. The creative judgment doesn't disappear — it moves upstream, into the system itself.01

Engineers build contracts.

Instead of building components from scratch, engineers build the contracts and infrastructure that let components be assembled correctly at scale: the props, the states, the diagnostic checks, the schemas.01

Handovers become stewardship.

The doer-to-manager shift means reviews can happen iteratively — even on a phone between meetings — and "handover" gives way to collective stewardship of the system itself.05 It is not less work. It is different work, and it requires more intentionality, not less.01

Interlude · 02

Risks & traps.

AI doesn't fix a design system; it amplifies whatever shape your system is already in. Six pitfalls the essays keep returning to.

  1. i

    Hallucinated files and ghost components.

    Agents will confidently reference components, props, and paths that do not exist. The fix is not "tell it to stop" — it is structured ground-truth (MCP, AGENTS.md, schemas) and stable, well-maintained sources to draw from.05

  2. ii

    Sameness creep.

    Everyone is using the same kits and the same "vibe codes". Fast remixing speeds delivery but stifles diversity — most teams still reinvent rather than extend, and end up with the same default aesthetic.05 Ask: where does our product actually need to look unlike the default, and have we documented that opinion clearly enough for an agent to hold the line?

  3. iii

    Bias and accessibility blind spots.

    Ethical checks for bias and inclusivity become more important once an agent is composing UI at speed.05 Accessibility requirements belong inside component contracts and decision-rule skills — not in a separate doc no agent reads.

  4. iv

    "It's multimodal — it can just look at the frame."

    Tempting, but only a partial fix. A screenshot closes part of the interpretation gap, but it is still inference from pixels, and it does nothing for drift — the screenshot is stale the moment the component changes. Vision patches reading; it does not patch maintenance.06

  5. v

    Letting the LLM write your documentation.

    The output reads like a third-rate style guide — "An input is an element users can type into" — and lacks the context your team actually needs about when and why to use a component.03 Use LLMs to migrate, to keep examples in sync, to write codemods. Don't use them to originate the system's voice.

  6. vi

    Treating "we have a design system" as a finish line.

    Design systems are crossing the same threshold responsive design crossed a decade ago. In two years, asking "do you have a design system?" will sound as quaint as "is your site mobile-friendly?" It is not a differentiator. It is the minimum cost of entry for participating in AI-generated, adaptive interfaces.04

The readiness checklist.

Tick what you can defend. Anything unchecked is where an agent will guess.

  • Every token is semantically named and used consistently in shipped code.
  • Every component has a structured contract — props, states, interactions, a11y — that a machine can parse.
  • Documentation lives in description fields as markdown, not as orphaned visual frames.
  • An AGENTS.md manifest sits at the root of the codebase.
  • An MCP (or equivalent) exposes the design system to agents as queryable structure.
  • Decision-rule skills capture the why: when to use which component, what to never do.
  • Patterns ship as schemas of relationships, not as monolithic code components.
  • A diagnostic layer reports drift between Figma source and production code.
  • Storybook (or equivalent) examples are kept in sync — automatically, not by memory.
  • The system extends an existing ecosystem where possible, and contributes back when it patches one.
