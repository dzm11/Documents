00 — Primer

Before you start.

If you've already shipped a design system and you've at least heard of MCP, you can skip this and jump to section 01. Otherwise — read this first; everything later assumes it.

Who this is for

  • DS practitioners — you own the system. You will implement most of this directly.
  • Designers using Figma daily — sections 03 and 05 are the core for you.
  • Engineers shipping with AI assistants — sections 02 and 04 describe the contracts your agents read from.
  • Curious newcomers — keep this primer open; come back to it when a term stops meaning anything.

Quick glossary — 8 terms used throughout

MCP
Model Context Protocol. A small server that hands an AI agent structured data about your system — so it can query the source of truth instead of scraping screenshots.
AGENTS.md
A manifest file at the root of a codebase. Tells an AI assistant which components to use, which to avoid, where the tokens live.
Skill
A small instruction file that encodes one decision rule for an agent. Example: "use the Switch component when only one option is selectable".
Declared docs
Component documentation written in description fields as markdown, so an agent can parse it without inferring meaning from a picture.
Drift
The gap between what was designed and what actually shipped. Usually invisible until production.
Diagnostic layer
Automated checks that detect and report drift between Figma source and production code.
Slot
A Figma feature. Creates a flexible content region inside a component, so the same component can hold different children without exploding into variants.
Code Connect
A Figma feature that explicitly links a design component to its production code counterpart — declaring how variants and props map.
01 — Introduction

Why this exists.

Design systems were always a means to an end. They existed to ship products faster and more consistently. The end has just changed shape: agents now assemble your product in the moment, from your component library — and the margin for error in that output is zero.01

A messy system used to be a productivity problem. Today it's a strategic liability. A human developer can route around inconsistent tokens or stale docs; an agent can't.04 This manual collects what to fix, in the order that matters, so an agent can compose your product without guessing.

1.1 — The shift

From human-readable to agent-readable.

The good news, repeated across every essay: the fix is unglamorous and well-known. Tokens. Contracts. Schemas. Machine-readable documentation. A diagnostic layer that keeps design and code in sync.01 Do that work in the right order and you don't just survive the shift — you get to use it.

1.2 — How models "see" Figma

Why Figma is harder for an AI than Paper.

Every Figma file is read by two audiences now. The designer next to you, and the AI model your team plugged in last month. When the file is prepared for both, agents generate faster, use fewer tokens, and hit your intent on the first try.

It's no accident that tools like paper.design and pencil.dev exist — and that paper's founders came out of Figma. Paper renders real HTML; Figma draws vectors. When Figma is your source of truth, the model has to interpret those vectors — and you pay for it in tokens, latency, and accuracy. The fix is to give the model the structure it actually needs.

Premise

Figma will remain the best tool for design work for the foreseeable future. The question is not whether to keep using it but how to make it cheap and accurate for an LLM to read.

The mental model that helps: when you connect Figma to your AI-powered IDE through MCP, you are turning your design file into a prompt.r1 Everything ambiguous about the file becomes ambiguous in the generated code.

02 — The foundations

Eight steps, in order.

The chain is unforgiving. You cannot generate adaptive UI without a system an agent can reliably consume; you cannot have such a system without a diagnostic layer that keeps design and code in sync; you cannot build that layer without first fixing the drift that caused it. Skip a step and the chain breaks.01

01 Posture · Foundation

Treat the system as an API, not a gallery.

An API doesn't hand you a picture and ask you to figure out what it meant. It returns structured data with a contract.06

The single most important shift: document your system as declared data, not as a designed artefact.

Designed frames — visual layouts, do/don't pairs, callout boxes pinned around a component — are optimised for human browsing. But they force an agent to infer meaning from a node tree.06 Declared documentation lives in description fields, in markdown. An agent can parse it without guessing.

  • Populate component description fields with clear markdown: purpose, usage, constraints, do/don't expressed as text.
  • Couple documentation to components — rename or delete the component and the docs travel with it.06
  • Keep designed frames for human stakeholders, but stop treating them as machine-readable.
  • The readability test: can an agent understand this without visual inference? If not, declare it.
02 Tokens · Contracts

Fix the foundation before you generate anything.

"Mostly documented" or "good enough for the team that built it" is not enough. Complete is the bar.01 For AI to generate a correct, branded, accessible interface, it needs absolute clarity about what your components are, how they behave, and the constraints they must respect.01

Failure
Where you pay
Hardcoded hex values in production code
Every brand tweak becomes a find-replace sweep.
Inconsistent token names across files
Agent picks the wrong one. Silently.
Missing component metadata
Agent infers behaviour from the name alone.
Component locked to one platform
Half a contract. Second platform reinvents it.01
  • Audit tokens: are they consistent, semantic, free of one-off hex values?04
  • Write component contracts that specify every prop, state, interaction pattern, and accessibility requirement.01
  • Add metadata that a machine can parse — not just human-readable Figma annotations.
Field note

Two real failure modes from enterprise teams: designers building careful Figma variants while engineers ship inline styles; and a 50-engineer team colour-picking from screenshots because the design-to-code path didn't exist.04 Same root cause both times: no structural communication encoded in the system itself.

03 Guidelines · Decisions

Encode the why, not just the how.

LLMs are trained on generic code and design practices. The choices already represented in your system — and not encoded anywhere — are exactly what they get wrong.02

Most component docs describe mechanics: what props exist, the difference between a checkbox and a radio. That is the how. What an agent needs in addition is the why: when to use a switch instead of a checkbox, when never to use a semantic-red chip (e.g. to indicate medical risk status), when a "select all / none" pattern is required.02

  • Treat design guidelines as a new deliverable, shipped alongside components and reference docs.02
  • Write them as agent skills: short, decision-oriented, product-specific.
  • Include negative examples — "do not generate X for Y context" — because the default training data will get this wrong.02
04 Guardrails · Tooling

Put MCPs, AGENTS.md, and skills in the codebase.

When LLMs have access to your codebase, they'll use anything they find as a pattern. The models aren't trained on your code; they're trained on public code — which is why Tailwind and ShadCN are so over-represented.02

  • MCP server — exposes the design system to agents as a structured resource they can query, not scrape.
  • AGENTS.md — a manifest at the repo root telling agents which components to use, which patterns are forbidden, where the tokens live.
  • Agent skills — small, focused instruction files that capture decision rules (the why from step three).02
# AGENTS.md (excerpt)
use_components_from: "@acme/ds"
forbid:
  - "inline tailwind utility classes for colour"
  - "recreating components that already exist in @acme/ds"
tokens: "./tokens/semantic.json"
skills: "./.skills/"
05 Patterns · Schemas

Ship components as data — not just code.

Patterns have always been the design-system bogeyman: built for one product purpose, made of many sub-components, exploding in variations. Encoding them as rigid code components creates maintenance burden and pushes teams toward workarounds.02 Schemas describe the relationship instead — which components compose into which patterns, what context they require, which siblings are valid. Less code; more contract.

Example · components-as-data
// A button described by its relationship, not its code.
{
  "name": "CancelButton",
  "extends": "Button",
  "context": "modal",
  "siblings": ["ConfirmButton"],
  "intent": "dismiss without action"
}
  • For each pattern, ship a schema describing its sub-components, contexts, and allowed siblings — not a monolithic code component.
  • Use semantic naming so an agent can compose meaningfully (a CancelButton in modalContext is far clearer than a generic Button).02
  • This is what enables multi-turn, dynamically recomposing UI — the same primitive that lets products like Expedia render inside ChatGPT.02
06 Diagnostic · Drift

Build the diagnostic layer.

No feedback loop. No diagnostic layer. Just two teams working in parallel, producing divergent results.04

Drift is the check-engine light of design systems. It rarely fails catastrophically; it just quietly diverges, sprint by sprint, demoralising the people who care most.04

  • Automated parity checks between Figma source-of-truth and production code (tokens, components, props).
  • Bidirectional sync — when a designer changes a variant, code knows; when an engineer ships a divergence, design sees it.
  • Treat drift as technical debt with a measurable cost, not a vibes problem.
07 Day-to-day

Put LLMs to work on the unglamorous tasks first.

Right now, the best way to use LLMs in your design system is to focus on meaningfully filling gaps. Think less about generating new components and more about delivering on the promise of design systems.03

  • Refactors and migrations — token renames, API changes, deprecations. Glorified find-and-replace at scale.03
  • Interaction prototyping — let designers exercise real flows in the browser via tools like v0, Lovable, or Dessn.03
  • Keeping examples fresh — Storybook stories and sandbox states that go stale the moment a component changes.03
Anti-pattern

Don't let an LLM write your design-system documentation. The output reads like a third-rate style guide ("An input is an element users can type into") and lacks the context your team needs about when and why to use a component.03

08 Posture · Ecosystem

Build in the open. Extend, don't restart.

Rule of thumb

Extend what already exists.

Contribute back when you patch it.

Start from scratch only when nothing else fits.

AI thrives in open environments because it has been trained on them. The corresponding warning is sameness — when every team uses the same kits and the same prompts, diversity collapses and the design system stops carrying any identity.05

  • Adopt or extend existing systems gradually, component by component.05
  • Make transparency and machine-readability architectural principles from day one.
  • Contribute back when you patch or extend — it keeps the ecosystem (and the models trained on it) honest.
  • Curate against vibe-coded sameness: where does your product actually need to look unlike the default?
03 — The design side

Making Figma legible to AI.

When Figma is connected to an editor through MCP, the model does not see pixels the way a human does. Everything that follows assumes Figma stays the design surface and that the goal is to encode as much intent into the file as possible — so the agent has less to interpret and more to consume.

3.1 — Inputs to the model

What an LLM sees through MCP.

Connect Figma to your editor via an MCP server and the model primarily receives these signals — not pixels.

  • Components & instances Names, descriptions, dev resources, props, variants.
  • Auto-layout Directions, gaps, padding — the visual cousin of flexbox; maps to your DOM structure.
  • Variables Colours, typography, spacing, code syntax mapping.
  • Text content & styles The actual strings plus the type tokens applied.
  • Page & frame structure Names matter — call frames Header, Hero, CTA, Row, Content; never Frame 2137.
  • Exportable assets Their names become file names downstream.
  • Annotations Anything you write into the file as structured intent.
  • Constraints For absolute-positioned and frame elements — how they should resize.

The takeaway: every layer name, every variable mapping, every annotation is doing real work. Whatever the model cannot read structurally, it has to guess at — and you pay for the guess in tokens, latency, and quality.

File hygiene checklist

  • Organise pages by intent — Style Guide, Components, Design screens — and keep exploration separate from production.r1
  • Delete or rename placeholder layers (Rectangle 23, Text 1, Frame 49). The model reads them and treats them as semantic.r1
  • Match component names to your intended code structure (Button / Primary, not Btn Blue).r1
  • Use realistic content instead of Lorem Ipsum so generated copy and layout feel real.r1

Responsive specifications

Create separate frames per breakpoint with explicit names so the agent gets responsive intent for free:r1

Home / Desktop / 1440
Home / Tablet / 768
Home / Mobile / 375
3.2 — Interaction

Share interaction intent, not just appearance.

Communicating interactive elements is as important as the static ones. Add annotations that capture desired behaviour — states, transitions, dynamic content. Clear notes on what should happen on hover, on focus, when data is empty, when an action is destructive — these reduce guesswork dramatically and push the first-pass output much closer to your intent.

A button without annotation is just a coloured rectangle to the model. A button annotated "Loading state shows spinner; disabled when form invalid; primary CTA on confirmation screen" is a contract.

3.3 — Source of truth

Variable code syntax.

Variable code syntax in Figma gives both designers and AI coding tools a single source of truth for colours, typography, spacing, and states. Every variable maps to its production-side counterpart — a Tailwind class, a CSS custom property, a design token name — and the model gets that mapping for free.

This is also something you can auto-generate from scripts or via the unofficial Figma Console MCP — meaning you don't have to type the mapping by hand for every token. Make it part of the file template so every new design starts with the mapping already in place.

Naming conventions worth committing to

Be intentional and semantic. The model uses these names verbatim downstream:r1

# colour
color/background/surface
color/text/primary

# typography
font/heading
font/body

# spacing
space/4
space/8
space/16

# radii & shadows
radius/sm   shadow/md
3.4 — The bridge

Code Connect & slots.

Code Connect explicitly links a Figma component to its production code component, documenting how variants and properties map between the two. It removes an entire class of "is this the same thing on both sides?" questions — and once an MCP server can read it, the agent inherits the mapping.

Slots are supported in Code Connect, and they appear directly in the rendered code snippets. When an engineer (or an agent) looks at a slot-based component in Figma, they immediately see how it maps to production: structure stays consistent, content stays flexible. This is exactly what container components need — the place where variation is expected by design.

Why it matters

Without Code Connect the agent has to infer the design ↔ code mapping every time. With it, the mapping is declared once and reused everywhere downstream — handoff becomes predictable instead of repeatedly negotiated.

Slot configuration patterns

Two intentional choices to make for every slot:r3

  • Pre-filled vs. empty. Default content provides context and clarity; an empty slot signals to the designer that they must add something. Pick one deliberately per slot — both are valid, neither is the default.
  • Preferred instances. Set the recommended component instances a slot can accept. They appear as curated options when designers interact with the slot, which accelerates the workflow and nudges everyone toward consistency without forcing it.

Implementation is light: insert a frame inside a main component, right-click → "Convert to slot" (⇧⌘S). Slots can't live on canvas directly, only inside a main component.r3

3.5 — Tooling

Internal plugins to speed up handoff.

The fastest wins are small Figma plugins tailored to your system. Run a brainstorming session with the design + DS team. A starting backlog:

  • Attach the correct variables to non-zero values: radius tokens to radii, spacing tokens to spacing.
  • Reconnect detached typography styles automatically.
  • Highlight elements that don't have a colour style applied.
  • Rename layers from Frame 2137 to something meaningful — div, stack, flex, card.
  • Scan all layers inside a given frame and report component-coverage percentage. Designs below 50 % go to the DS team for review.
  • Monthly Library Analytics digest — which team has lowest coverage, which component is most often detached.

Each one of these is small. Together they remove an enormous amount of friction from the file before anything ever reaches a coding agent.

3.6 — Drift detection

Audit alignment with the Figma MCP server.

With the Figma MCP server an agent can use design context to audit drift between your designs and codebase — and propose structural fixes on either side:

  • Audit design-token usage in code vs. the selected Figma frame, and vice versa.
  • Flag layer names that should change to better align Figma with code components.
  • Suggest Figma prop changes that bring the design closer to the production API.
  • Make code responsive and surface design issues earlier in the cycle.

Example prompt — feed it the Figma selection and the production files, ask for divergences:

#get_variable_defs from the selected Figma elements and compare them
with the variables and styles used in the code implementation in
#file:ChatBox.tsx and #file:chatBox.css
04 — The code side

Testing & verifying what was generated.

Once an agent has produced code from your system, you need a way to verify it without sitting through a manual QA pass on every output. Storybook is the cheapest answer.

4.1 — Verification

Storybook as test runner.

Storybook is an open-source tool that lets developers build, test, and visualise UI components in isolation. Its built-in test runner — based on Jest and Playwright — automatically verifies whether a component renders correctly, while also validating accessibility and performance characteristics. Visual-regression testing can run in the cloud to catch issues caused by CSS changes.

For an agent-driven workflow this is essential: every component generated or modified by an LLM should land in a story that exercises its states, and the test runner becomes your safety net. Without it, drift between generated code and intent is invisible until production.

Pairing

Step 7 ("LLMs on unglamorous work") and Storybook reinforce each other: ask an agent to keep stories in sync with components, then let the test runner catch any case where it didn't. The two combined remove the single most tedious DS maintenance task.03

05 — Roles

The new craft.

If agents can compose interfaces, what happens to the people who used to compose them? The honest answer from across the essays: the work doesn't shrink, it relocates — and at the same time, an entirely new role is emerging in its own right.0105

5.1 — Relocation

The roles move upstream.

From

Building individual screens, individual components, individual stories. Picking colours, drawing variants, shipping pixel mocks across to engineering. Performing repetitive maintenance by hand.05

→
To

Authoring the tokens, contracts, and decision rules that the system uses to assemble itself. Defining the vocabulary, not arranging the sentences. Reviewing, refining, and governing what agents produce.0105

Designers become rule-authors.

Instead of designing individual screens, designers define the vocabulary, constraints, and rules that govern how screens get generated.01

Engineers build contracts.

Instead of building components from scratch, engineers build the contracts and infrastructure that let components be assembled correctly at scale.01

Handovers become stewardship.

The doer-to-curator shift means reviews can happen iteratively — even on mobile — and "handover" gives way to collective stewardship of the system itself.05

5.2 — A new role

The Design Engineer.

A new role gaining ground — both confusing and exciting, because expectations for how software should look and feel have never been higher. Design Engineers are the practitioners who close the gap between intent and implementation, and they are increasingly the people whose work makes an AI-coded interface feel considered rather than generic.

The Design Engineer's toolbox

  • Design in Figma — fluent, but not exclusively
  • Design in code — comfortable shipping production-quality interfaces
  • Debug browser performance — and care about it
  • Design micro-animations and motion details
  • Create and maintain core components on both sides (design + code)
  • Maintain documentation — Storybooks, READMEs, MDX
  • Improve accessibility — and verify it, not just assert it

What they care about

For the web, the Design Engineer's priorities tend to include:

  • User interactions — fidelity, responsiveness, feel
  • Reusable components and primitives
  • Design and page-speed quality together, never one without the other
  • Inclusive input modes — touch, pointers, keyboard
  • Respecting user preferences — reduced motion, colour scheme, language
  • Accessibility for assistive-technology users

There's a lot of work behind the pretty pixels. — operating principle

Design Engineers go beyond visual appeal. They own the pieces that make an interface feel considered — motion, performance, accessibility, micro-detail. In an agent-driven workflow they end up running the system itself: authoring the contracts, writing the skills, maintaining the stories, reviewing what the agent produced.

How Vercel describes the role

Vercel's own framing of the role is worth quoting almost verbatim — it covers the three working modes a Design Engineer moves between:r7

  • Design collaboration. Partnering with designers iteratively in Figma or in code — consulting on implementation cost, component reusability, and prototyping explorations that are easier to execute outside design tools.
  • Product team integration. Embedding within a product team for month-plus features, implementing UI while platform engineering handles the infrastructure.
  • Independent ownership. Owning small-to-medium features end-to-end — from sketching through shipping — and driving design-led system projects (the Geist font, the documentation site, the component library).

Vercel emphasises a culture point that travels well: focus on the outcome, not the process or the tool — and "iterate to greatness" rather than wait for perfection.r7

06 — Risks & traps

Where this goes wrong.

AI doesn't fix a design system; it amplifies whatever shape your system is already in. Six pitfalls the essays keep returning to.

i

Hallucinated files and ghost components.

Agents will confidently reference components, props, and paths that do not exist. The fix isn't "tell it to stop" — it's structured ground truth (MCP, AGENTS.md, schemas) and stable, well-maintained sources to draw from.05

ii

Sameness creep.

Everyone uses the same kits and "vibe codes". Fast remixing speeds delivery but stifles diversity — most teams still reinvent rather than extend, and end up with the same default aesthetic.05

iii

Bias & accessibility blind spots.

Ethical checks for bias and inclusivity become more important once an agent is composing UI at speed.05 Accessibility requirements belong inside contracts and skills — not in a separate doc no agent reads.

iv

"It's multimodal — let it look at the frame."

A screenshot closes part of the interpretation gap, but it's still inference from pixels, and it does nothing for drift — the screenshot is stale the moment the component changes. Vision patches reading; it doesn't patch maintenance.06

v

Letting the LLM write your documentation.

The output reads like a third-rate style guide and lacks the context your team actually needs about when and why to use a component.03 Use LLMs to migrate, sync stories, write codemods — not to originate the system's voice.

vi

"We have a DS" as the finish line.

Design systems are crossing the same threshold responsive design crossed a decade ago. In two years, "do you have a design system?" will sound as quaint as "is your site mobile-friendly?" — minimum cost of entry, not differentiation.04
